So you want to hack on NetworkManager?

First, join the mailing lists: http://mail.gnome.org/mailman/listinfo/networkmanager-list.

Second, read the Design Goals of NetworkManager to get an idea of why things work the way they do.

Next, grab NetworkManager from CVS HEAD. To checkout and build from CVS HEAD:
cvs -d :pserver:anonymous@anoncvs.gnome.org:/cvs/gnome login
cvs -d :pserver:anonymous@anoncvs.gnome.org:/cvs/gnome co NetworkManager
./autogen.sh --prefix=/usr --sysconfdir=/etc --localstatedir=/var
make
make install

Actually hacking on it

If you make changes to NetworkManager, create the patch using a Unified diff format (ie, diff -u) and mail the patch to the NetworkManager list. Others on the list may make suggestions, please take these into consideration. Eventually, your patch can be checked into NetworkManager CVS.

Stuff to work on

  • WPA support: please see this message from the mailing list for a general idea of how WPA support should get integrated. The general idea first to simply spawn a new instance of wpa_supplicant when NM connects to an AP that needs WPA, generating a config file on-the-fly. Normal WEP should still be done by NM. Eventually, we want to talk to wpa_supplicant over its control socket like the wpa_cli program does.
  • libnl support: Check out libnl. Using it rather than callouts to /sbin/ip would allow us to get rid of most of the code in the system-specific backends. Some of this has already been prototyped, and it works just fine. This should be pretty easy.
  • Devices as GObjects: though NetworkManager is written in C, it could benefit from some object-oriented features that glib provides. The code would be a lot cleaner and more exentsible to things like Bluetooth devices if the NMDevice object was GObject-derived. Specific features of wireless, wired, etc devices could be exposed through subclasses of a GObject-derived NMDevice.
