Things we plan to do in Gnumeric

These are our immediate (and not-so-immediate) plans for new features or just work to be done in Gnumeric. Feel free to contact us if you want to help or just discuss any of these.

The developers' scratch pad gives an idea of which tasks are outstanding on various topics.

Immediate plans

  • MS Excel 95 image import A patch is in bugzilla but was too invasive to go into 1.4.0.
  • Any Release Critical reports The BUGS file keeps a list of known release problems and plans at the top.

Intermediate plans

  • Right to Left. Work is mostly complete to support right to left layouts. The status is detailed in the status file and the current patch is in bugzilla.
  • Function documentation reoganization. The current documentation for functions like SUM, VAR, and so on is one large string per function. That makes life hard on translators and invites large amounts of duplication. An Overview of the proposed system is in the mailing list archive.
  • Pivot tables. I'm going to start on this over the holidays (dec 2004). We have docs (from the MS Excel 97 developers kit) of the records. Now I just need to get a feel for what these do.
  • libgoffice and libgnumeric. Work is underway in cvs to finish the split of libgoffice into a stand alone package. The status is kept in the goffice_split branch of cvs. This will make the charting engine, and powerpoint viewer available without linking to gnumeric itself.

Wish-list (long-range plans)

  • Scripting. After some vacilation trying to decide between wrapping Gnumeric's object model, and providing something that is VBA compatible, the latter won. We'll export an api in C that is compatible with the widely used VBA interface, and wrap it in several languages (Python, VB.NET, and C# are likely candidates).
  • Format engine rewrite. Our current number formatting and matching engines are based directly on Excel format strings. That is truly sub-optimal and needs to be rewritten in terms of sensible objects.
